## ② 安全审计与修复

### 修复清单（本次已改）

| 级别 | 问题 | 位置 | 修复 |
|------|------|------|------|
| 🔴 严重 | **简历接口鉴权形同虚设**：拦截器写 `/resume/**`，控制器实际是 `/resumes`，模式不匹配→所有简历接口免登录可访问 | `WebMvcConfig` | 路径改 `/resumes/**`，并排除公开分享 `/resumes/share/*` |
| 🔴 严重 | **水平越权（IDOR）**：`userId` 由 `@RequestParam`/请求体传入，传 `?userId=N` 即可读写他人简历 | `ResumeController` 等 | 新增 `@CurrentUserId`，userId 统一取自 **Session**，客户端无法伪造 |
| 🟠 高 | `versions`/`restore`/`share`/`getShare` **完全无归属校验**，凭 resumeId 即可操作任意简历 | `ResumeController` | 增加 `notOwned()` 归属校验 |
| 🟠 高 | 面试记录、会员兑换/流水 **免登录 + IDOR** | `InterviewController`、`MemberController` | 纳入拦截 + `@CurrentUserId`；提交以登录态覆盖请求体 userId |
| 🟡 中 | **上传后缀取自用户文件名**：`image/png` 内容配 `evil.html` 后缀→落地为可被当 HTML 解析的文件（存储型 XSS） | `FileUploadServiceImpl` | 扩展名改由**已校验的 contentType** 推导；上传纳入鉴权 |
| 🔴 严重 | **社区审核/删除接口无鉴权**：`approve`、`/community/admin/*`、删除对匿名开放，任何人可审核/删除任意内容 | `CommunityController` | `approve`/后台列表限管理员；删除限**管理员或作者本人**；按简历删投稿仅限本人 |
| 🟠 高 | **社区投稿/点赞、模板收藏 IDOR**：`userId` 由客户端传入 | `Community`/`TemplateController` | 改 `@CurrentUserId`（会话注入），变更类操作要求登录 |
| 🟡 中 | **会话安全**：Cookie 无 `HttpOnly`/`SameSite`，无会话固定防护 | `application.yml`、登录控制器 | Cookie 设 `http-only`+`same-site=lax`（secure 生产开）；登录后 `changeSessionId()` 防固定 |
| 🟡 中 | **`ownedBy` 把 `userId==null` 当演示用户 1** | `InMemoryDataRepository` | null 不再归属任何简历（未登录读/删全空），保留历史 owner 空→1 兼容 |

新增机制 `@CurrentUserId`（`config/CurrentUserId.java` + `CurrentUserIdArgumentResolver.java`）：从服务端 Session 注入登录用户 ID，是消除这一类越权的统一手段。

### 行为变化（需注意）

简历、面试等接口现在**必须登录**（修复前因路径 bug 实际不需要）。前端是 Session 登录、且会带 Cookie，登录后正常；**未登录的「演示模式」直接访问这些接口会返回 401** —— 这是正确的安全姿态，但若你依赖免登录演示，请确认。

此外，**社区投稿/点赞、模板收藏现在也要求登录**（未登录返回「请先登录」）；社区案例/文章/模板的**浏览**仍公开。前端调用里残留的 `userId` 入参已被后端忽略（改用会话），无需立即改但建议后续清理。

| | 改造前 | 改造后（着陆/首页首屏） |
|---|---|---|
| 入口 JS | 单包 **2502 KB / gzip 835 KB**（含 echarts，全部首屏加载） | index+vue-core+vendor+element-plus+Home ≈ **gzip ~378 KB** |

| 其余页面 | 全在首包 | 各自独立 chunk，按路由加载 |

首屏 JS 体积约 **-55%**，最大的 echarts 完全延后。

**可进一步优化（未做，已记录）**：`main.js` 是 Element Plus 全量引入，无法摇树。引入 `unplugin-vue-components` 按需导入可再砍掉一大块 element-plus 体积（改动较大，留作后续）。

## ④ 编辑器 v-if 链健壮性梳理

**起因**：本次最初修复的「图标文字重复渲染」就是一条类型分发链被中途插入的 `v-if`（字数统计 span）截断，导致兜底 `v-else` 错配、内容渲染两遍。

**梳理范围与结论**：

| 组件 | 链类型 | 结论 |
|------|--------|------|
| `DragResumeCanvas` | 组件类型分发（13 分支 + 编辑态 + 兜底） | 唯一出问题处，**已修**：字数统计 span 移到 `v-else` 之后、独立 `v-if`，链恢复连续；并加注释防回归 |
| `TemplatePreview` | 同类型分发 | 干净，单链以 `v-else` 兜底，无中断 |
| `ResumeVisual` | 图表类型分发（radar→…→statcard） | 干净，同级连续；内部 `v-if` 均嵌套在各分支内、作用域正确；无 `v-else` 无错配 |
| `InterviewView` | 面试阶段分发（intro→running→loading→report） | 干净，顶层连续；各 `v-else` 均嵌套在分支内、与相邻 `v-if` 正确配对 |
| `EditorView`/`ProfileView` 等 | 布尔开关（X / 非X） | 非分发链，逐一核对配对正确 |

**全局佐证**：`vite build` 无任何「`v-else`/`v-else-if` 缺少相邻 `v-if`」告警 → 不存在被普通元素截断的链。

**沉淀的规范**：类型/状态分发链里，**不要把独立用途的 `v-if` 元素插在最后一个 `v-else-if` 与兜底 `v-else` 之间**；这类浮层（字数、徽标等）应放在 `v-else` 之后并自带独立 `v-if`。该约定已以注释写入 `DragResumeCanvas`。
